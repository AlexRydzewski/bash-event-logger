# Event Control and Logging Subsystem

**Version:** 1.0.0  
**Released:** 2026-05-14  
**License:** [MIT](LICENSE)

## Purpose

The `__event_control()` function implements a centralized event-processing,
logging, verbosity, and tolerance mechanism for shell scripts.

It is intended to:

- unify message handling,
- centralize exit logic,
- support both interactive and non-interactive execution,
- integrate with syslog/journald,
- optionally log into private files,
- provide configurable verbosity and fault tolerance levels.

The subsystem behaves similarly to a lightweight logging framework adapted for
Bash scripting.

---

# Global Initialization

```bash
readonly PROG_NAME="$(basename -- "$0")"
readonly PID=$$
```

## PROG_NAME

Stores the current script basename.

Used for:

- logger tags,
- log formatting,
- identification in syslog/journal.

Example:

```text
backup.sh
```

---

## PID

Stores current process ID.

Used for:

- logger identification,
- correlating log entries,
- multi-instance diagnostics.

---

# Core Runtime Controls

```bash
: "${TOLERANCE:=3}" "${VERBOSITY:=5}"
```

This initializes runtime defaults only if variables were not already provided
from the environment or caller.

---

## VERBOSITY

Controls which messages are visible/logged.

Higher values produce more output.

| Level | Meaning |
|---|---|
| 0 | Silent |
| 1 | Fatal |
| 2 | Critical |
| 3 | Error |
| 4 | Concern |
| 5 | Warning |
| 6 | Info |
| 7 | Operation start |
| 8 | Success |
| 9 | Debug |

---

## TOLERANCE

Controls which events terminate execution.

Lower event levels are considered more severe.

| Event Level | Behavior |
|---|---|
| `< TOLERANCE` | terminate script |
| `>= TOLERANCE` | continue execution |

Default:

```bash
TOLERANCE=3
```

Meaning:

- levels `0-2` terminate,
- levels `3-9` continue.

---

# Event Level Model

The function combines:

- logging,
- verbosity filtering,
- execution control,
- exit status handling.

Conceptually:

```text
Debug        Info        Warning        Error        Silent
----------------------------------------------------------------
9 8          7 6         5 4 3          2 1          0
```

---

# Typical Semantic Usage

| Level | Meaning |
|---|---|
| 0 | silent successful stop |
| 1 | unrecoverable fatal error |
| 2 | critical failure |
| 3 | severe error |
| 4 | degraded but recoverable state |
| 5 | warning |
| 6 | function entry |
| 7 | operation entry |
| 8 | successful operation |
| 9 | debug/internal state |

---

# Special Level 0

```bash
[[ ${2:-${TOLERANCE:-3}} -eq 0 ]] && local status=0
```

Level `0` is a special case:

- execution stops,
- exit status becomes `0`,
- intended for silent/non-error termination.

This allows implementing controlled early exits without reporting failure.

---

# Verbosity Filtering

```bash
[[ ${2:-5} -gt ${VERBOSITY:-5} ]]
```

If the event level exceeds current verbosity:

- message output is suppressed.

However termination logic still applies.

This is important because:

- fatal conditions must still terminate,
- even if logging is disabled.

---

# Termination Logic

Core decision:

```bash
[[ ${2:-3} -ge ${TOLERANCE:-3} ]] && return
exit
```

Meaning:

| Condition | Result |
|---|---|
| level >= tolerance | continue |
| level < tolerance | terminate |

This creates a simple severity threshold model.

---

# LOG_FILE Handling

If `LOG_FILE` is defined:

```bash
[[ -n "${LOG_FILE:-}" ]]
```

the function switches to file logging mode.

---

## Directory Auto-Creation

If target directory does not exist:

```bash
mkdir -p "$(dirname -- "$LOG_FILE")"
```

Behavior depends on tolerance:

| Tolerance | Behavior |
|---|---|
| `>=3` | attempt auto-create |
| `<3` | treat as fatal |

---

# Secure Log File Creation

```bash
( umask 077 && : >>"$LOG_FILE" )
```

This:

- creates file if absent,
- preserves existing file otherwise,
- ensures restrictive permissions.

Resulting permissions are typically:

```text
600
```

---

# Why `:` Is Used

```bash
: >>"$LOG_FILE"
```

`:` is POSIX no-op builtin.

It is used only to attach redirection with minimal overhead.

Equivalent alternatives:

```bash
true >>"$LOG_FILE"
```

or

```bash
echo -n >>"$LOG_FILE"
```

but `:` is lighter and cleaner.

---

# Log File Format

File entries are written as:

```text
May 14 12:00:00 host script[1234]: message
```

Generated by:

```bash
echo "$(date "+%b %d %H:%M:%S") ${HOSTNAME:-localhost} ${PROG_NAME}[$PID]: $1"
```

This intentionally resembles syslog formatting.

---

# Interactive Terminal Behavior

If:

- no `LOG_FILE`,
- stdout attached to terminal,

then logs are emitted to `stderr`:

```bash
1>&2
```

This preserves clean stdout for:

- pipelines,
- structured output,
- command substitution.

---

# Syslog / Journald Integration

Non-interactive execution defaults to:

```bash
logger
```

This integrates naturally with:

- syslog,
- journald,
- centralized logging systems.

---

# logger Version Detection

```bash
logger --version
```

is checked against:

```text
2.27.0
```

because newer util-linux versions support:

```bash
logger --id
```

which improves journald integration.

---

# Root vs Non-Root Logger Behavior

## Root

```bash
logger --id=$PID --tag=$PROG_NAME
```

Produces native structured journal records.

---

## Non-root

Fallback:

```bash
message_prefix="($PID) "
```

because unprivileged users may not be allowed to spoof PID metadata.

---

# WHY WHOAMI Uses `id -un`

```bash
WHOAMI=$(id -un 2>/dev/null || echo unknown)
```

`id -un` is preferred over:

```bash
whoami
```

because:

- it is more portable,
- directly tied to effective UID,
- avoids external utility dependency in some environments.

---

# Parent Shell Safety

The code intentionally resets:

```bash
unset message_prefix
unset logger_opts
```

to avoid inheriting stale shell variables from:

- parent shells,
- sourced environments,
- wrapper scripts.

---

# Startup Event Example

```bash
__event_control "The script is running as '$WHOAMI' with PID '$PID'" 5
```

Typical output:

```text
May 14 12:00:00 backup.sh[1234]: The script is running as 'root' with PID '1234'
```

---

# Intended Design Goals

The subsystem attempts to provide:

- centralized event handling,
- configurable fault tolerance,
- unified logging semantics,
- clean stdout/stderr separation,
- syslog compatibility,
- safe file logging,
- minimal external dependencies,
- POSIX-oriented behavior where practical.

It effectively acts as a lightweight operational framework for larger Bash
scripts.
